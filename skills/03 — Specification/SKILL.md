# Skill 03 - Specification

## Tujuan

Menyusun Software Requirements Specification (SRS) berdasarkan hasil elicitation sehingga menghasilkan Functional Requirement, Non-Functional Requirement, Business Rule, User Story, dan Acceptance Criteria.

---

## Kapan Digunakan

Digunakan setelah proses elicitation selesai.

---

## Input

- CASE.md
- docs/requirements/01-inception.md
- docs/requirements/02-elicitation.md

---

## Langkah Kerja

1. Baca seluruh dokumen requirement sebelumnya.
2. Susun Functional Requirement.
3. Susun Non-Functional Requirement.
4. Susun Business Rule.
5. Buat User Story.
6. Buat Acceptance Criteria.
7. Pastikan seluruh requirement memiliki ID.
8. Periksa konsistensi dokumen.

---

## Output

docs/requirements/03-specification.md

---

## Aturan

- Gunakan ID Requirement.
- Jangan menambah fitur di luar scope.
- Acceptance Criteria harus dapat diuji.

---

## Quality Check

- Functional Requirement lengkap.
- User Story sesuai stakeholder.
- Acceptance Criteria jelas.
- Tidak ada requirement yang bertentangan.

---

## Kondisi Gagal

- Input tidak lengkap.
- Scope belum jelas.

---

## Human Review

Periksa apakah seluruh requirement sudah sesuai kebutuhan stakeholder.
